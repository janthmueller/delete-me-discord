This page walks through a first cleanup preview from discovery to execution.

This page assumes you already:

- installed the CLI
- authenticated with `dmd login` or `DISCORD_TOKEN`

## 1. Discover IDs first

```bash
dmd list guilds
dmd list channels
```

This shows the IDs you can pass to `--include`.
IDs can refer to guilds, categories, supported message channels, thread parents, or threads.

All accessible active and archived threads are included by default. Thread discovery may require one or more paginated search requests for every thread-capable channel in every visible guild.

To inspect only active threads during an initial run, use:

```bash
dmd list channels -x archived
```

To exclude an exact thread type, use a canonical type name from the channel tree:

```bash
dmd list channels -x PrivateThread
```

Use `dmd list channel-types` and `dmd list thread-states` to inspect all accepted values. Use `-x threads` to omit every thread type.

Channel listing and cleanup include archived threads by default. To preview
only archived content, use:

```bash
dmd clean \
  --include <id> archived \
  --dry-run
```

See [Archived Thread Cleanup](../../guides/archived-thread-cleanup/) before a
broad run that may temporarily change thread state.

## 2. Start with dry-run

```bash
dmd clean --include <id> --dry-run
```

`--dry-run` makes `dmd` fetch and evaluate content without deleting messages, reactions, or thread containers.
Use it to check the selected targets and retention rules before running the same command for real.

Thread containers are not deleted in the default mode. Do not add `--delete-owned-threads` during a first broad run. That option removes an entire shared thread and belongs in a narrow thread-ID dry-run after you have read [Delete Owned Threads](../../guides/delete-owned-threads/).

`--include` accepts one or more IDs and filter values.
Omit it to use the default scope: all eligible channels and accessible threads the tool can see.

By default, the run also removes your reactions and preserves nothing because:

- `--keep-last` defaults to `0`
- `--keep-within` defaults to `0`
- reactions are deleted unless you opt out with `--keep-reactions`

Dry-run reports per-channel planning details such as:

- messages and reactions that would be deleted
- `scan time`
- `est. execute time`
- `est. total time`

Check that the output matches your intent:

- the correct channel name
- a believable number of messages or reactions in scope
- a per-channel summary that matches your expectation
- no surprise channels in the output

## 3. Adjust the command

Common adjustments:

- Add `--keep-within 2w` if you want to protect recent history.
- Add `--keep-last 20` if you always want to keep the last N messages in a channel.
- Add `--fetch-within 2w1d` later if repeated runs should only inspect a recent slice.
- Add `--max-messages 100` if you only want to inspect a small recent slice first.
- Add `-x archived` when cleanup should search active threads only.
- Store `exclude_thread_states=archived` in a reusable profile for the same active-only policy.
- Add `-x PrivateThread` to omit private threads, or use another displayed channel type such as `GuildVoice`.

Discord's application API documentation permits message deletion in archived
threads, but live user-account endpoints rejected direct mutations. DMD does
not mutate an archived thread until it has built a buffered plan with eligible
work. It then activates the thread and always attempts to restore its archived
state. Add `--skip-unrestorable-threads` when DMD should skip a thread unless
restoration rights can be reasonably established before cleanup.

## 4. Execute when the plan looks right

Drop `--dry-run` once you are satisfied with the output:

```bash
dmd clean --include <id>
```

If you want repeated cleanup over time, continue with [Rolling Retention](../../guides/rolling-retention/).
