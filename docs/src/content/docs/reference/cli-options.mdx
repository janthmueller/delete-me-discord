This page is the main reference for the current CLI surface.

For accepted value formats such as time deltas, ranges, booleans, and profile `--set` values, see [Value Formats](../value-formats/).

## Command Tree

The primary cleanup command is:

```bash
dmd clean ...
```

Other top-level commands are:

- `dmd login`
- `dmd logout`
- `dmd whoami`
- `dmd list guilds`
- `dmd list channels`
- `dmd list profiles`
- `dmd list channel-types`
- `dmd list thread-states`
- `dmd cache clear`
- `dmd profile fields`
- `dmd profile show`
- `dmd profile add`
- `dmd profile update`
- `dmd profile remove`

## Cleanup Command

### Scope and safety

| Short | Long | Default | Description |
| --- | --- | --- | --- |
| `-d` | `--dry-run` | off | Simulate deletions without making changes. Reports foreign reactions affected by parent-message deletion and, when owned-thread deletion is enabled, scans those threads to report container impact. Uses a separate `.dryrun.json` preserve cache file when preserve-cache is enabled. |
| `-i` | `--include SELECTOR...` | all eligible | Include complete Discord IDs, canonical channel types, `threads`, `active`, or `archived`. `--include-ids` remains an alias. |
| `-x` | `--exclude SELECTOR...` | none | Exclude complete Discord IDs, canonical channel types, `threads`, `active`, or `archived`. `--exclude-ids` remains an alias. |
| `-` | `--exclude-channel-types TYPE...` | none | Exclude exact message-bearing Discord channel types such as `GuildVoice` or `PrivateThread`. |
| `-` | `--exclude-thread-states STATE...` | none | Exclude `active` or `archived` threads. |
| `-` | `--exclude-threads` | off | Exclude all announcement, public, and private threads. |
| `-` | `--delete-owned-threads MODE` | `none` | Optionally delete creator-owned thread containers. `self-only` requires a complete scan with no messages from other authors; `all` permits deleting their messages too. Both are destructive and require `MANAGE_THREADS`. |
| `-` | `--skip-unrestorable-threads` | off | Skip archived threads when restoring their archived state cannot be reasonably guaranteed. |
| `-` | `--profile` | unset | Load cleanup defaults from a named profile in `config.json`. Explicit CLI flags still win. |
| `-t` | `--token` | unset | Override stored credentials and `DISCORD_TOKEN` for this command only. |

### Retention

| Short | Long | Default | Description |
| --- | --- | --- | --- |
| `-n` | `--keep-last` | `0` | Keep the last N messages in each channel. |
| `-` | `--keep-last-scope` | `all` | Count `keep-last` against all recent messages in the channel (`all`) or only your deletable messages (`mine`). |
| `-k` | `--keep-within` | `0` | Keep messages and reactions newer than this time delta. |
| `-` | `--keep-reactions` | off | Keep your reactions instead of deleting them during cleanup. |
| `-` | `--preserve-cache` | off | Persist kept message IDs between runs. |
| `-` | `--preserve-cache-path` | `~/.config/delete-me-discord/preserve_cache.json` | Override the preserve cache path. |

### Fetch and execution

| Short | Long | Default | Description |
| --- | --- | --- | --- |
| `-f` | `--fetch-within` | unlimited | Fetch only messages newer than this time delta. |
| `-m` | `--max-messages` | unlimited | Maximum number of messages to fetch per channel. |
| `-` | `--buffer-per-channel` | off | Buffer one channel at a time before evaluation. |
| `-` | `--fetch-sleep-time` | `0.2 0.4` | Minimum interval between message fetch requests. One value is fixed, two define a random range in seconds. |
| `-` | `--delete-sleep-time` | `1.5 2` | Minimum interval between cleanup mutation requests, including temporary thread-state updates. One value is fixed, two define a random range in seconds. |

### Timing and retries

| Short | Long | Default | Description |
| --- | --- | --- | --- |
| `-` | `--max-retries` | `5` | Maximum retry attempts for retryable API requests. |
| `-` | `--retry-safety-jitter` | `0.1 0.3` | Small jitter added to server-provided retry delays. One value is fixed, two define a random range in seconds. `--retry-time-buffer` remains a compatibility alias. |
| `-` | `--request-interval` | policy defaults | Override a named minimum interval as `POLICY=MIN` or `POLICY=MIN,MAX`. Repeat for multiple policies. |

DMD tracks Discord's global, route, bucket, and route-family rate-limit state in memory for the duration of a run. It learns from response headers and `429` payloads, then waits immediately before the next affected request. A user-scoped `429` can teach the scheduler a minimum interval shared by the same normalized endpoint across different channel or guild IDs. Fetch and delete intervals use the same scheduler, so there is no unconditional sleep after the final request in a sequence.

When Discord provides multiple relative retry deadlines, DMD uses the longest valid `retry_after`, numeric `Retry-After`, or `X-RateLimit-Reset-After` value and then adds `--retry-safety-jitter`. Standard HTTP-date `Retry-After` and absolute `X-RateLimit-Reset` values are accepted only when no relative deadline exists, avoiding dependence on local clock accuracy. If a network failure, HTTP `408`, server error, indexing response, or `429` contains no usable deadline, DMD uses capped exponential backoff with full jitter instead. A hintless user-scoped `429` blocks the affected route family for that retry without teaching a permanent interval from an unknown limit.

Named policy defaults:

| Policy | Default interval |
| --- | --- |
| `read` | `0.1 0.25` |
| `fetch` | `0.2 0.4` |
| `delete` | `1.5 2.0` |
| `thread-search` | `0.2 0.4` |

For example:

```bash
dmd list channels \
  --request-interval thread-search=0.5,0.8
```

An explicit `--request-interval` takes precedence over the corresponding `--fetch-sleep-time` or `--delete-sleep-time` compatibility option for that run.

### Output

| Short | Long | Default | Description |
| --- | --- | --- | --- |
| `-q` | `--quiet` | off | Only show warnings and errors. |
| `-v` | `--verbose` | `0` | Increase output detail. Repeat for more verbosity. |
| `-j` | `--json` | off | Emit JSON output for logs and discovery output. |
| `-r` | `--redact-sensitive [SUFFIX]` or `[PREFIX SUFFIX]` | off | Redact sensitive values in logs and discovery output. |
| `-` | `--redact-names` / `--no-redact-names` | on | Redact human-readable names when sensitive redaction is enabled. |
| `-V` | `--version` | n/a | Show the version and exit. |

## Discovery Commands

### Guild listing

`dmd list guilds` lists guild IDs and names available to the authenticated user.

### Channel listing

`dmd list channels` lists supported channels and all accessible active and archived threads, grouped by guild/category/parent plus DMs.

<Aside type="note" title="Thread discovery is the v3 default">
Starting with v3, `dmd list channels` searches every relevant text, announcement, forum, and media parent for accessible active and archived threads. Results are paginated and Discord may need to build a search index, so a complete listing can take noticeably longer than it did before v3.

When thread information is not needed, skip every thread search:

```bash
dmd list channels --exclude-threads
```

`dmd clean` uses the same active-and-archived default. Archived threads are
evaluated while archived and activated only when their cleanup plan contains
work. Use `-x archived` for active-only cleanup.
</Aside>

Use `--include archived --dry-run` to preview only archived threads. See
[Archived Thread Cleanup](../../guides/archived-thread-cleanup/).

### Profile listing

`dmd list profiles` lists the configured cleanup profile names from `config.json`.

### Filter value listings

`dmd list channel-types` lists the canonical values accepted by `--include` and `--exclude`.

`dmd list thread-states` lists the thread-state values accepted by `--include` and `--exclude`.

Both commands are local and do not require a Discord token. Add `--json` for a JSON array.

`dmd list guilds` and `dmd list channels` support:

- `--token`
- `--config-path`
- `--include`
- `--exclude`
- `--json`
- `--quiet`
- `--verbose`
- `--redact-sensitive`
- `--redact-names` / `--no-redact-names`
- retry flags

`dmd list channels` also supports `--exclude-channel-types`, `--exclude-thread-states`, and `--exclude-threads`.

`--include` and `--exclude` classify ASCII decimal values as complete Discord
IDs. Guild and DM IDs are validated from the current top-level lists;
category, channel, thread-parent, and thread IDs use an exact channel lookup.
Old suffix shorthand is treated as an exact ID and fails when Discord cannot
resolve it.

Canonical channel types are case-sensitive. `threads` groups
`AnnouncementThread`, `PublicThread`, and `PrivateThread`; `active` and
`archived` select thread states:

```bash
# Only active threads.
dmd clean -i active --dry-run

# All accessible public threads, including archived ones.
dmd clean -i PublicThread --dry-run

# One category, without voice or stage chat.
dmd clean -i <category-id> -x GuildVoice GuildStageVoice --dry-run

# Exclude all threads except one exact thread.
dmd clean -x threads -i <thread-id> --dry-run

# Exclude the thread group, but retain its narrower public-thread type.
dmd clean -x threads -i PublicThread --dry-run

# Include the thread group, except its narrower public-thread type.
dmd clean -i threads -x PublicThread --dry-run
```

Selector conflicts are order-independent and use specificity:

1. exact channel or thread ID
2. concrete channel type such as `PublicThread`
3. the `threads` type group

A narrower selector overrides a broader one. At equal specificity, exclusion
wins. Therefore `-i PublicThread -x threads` selects public threads, while
`-i PublicThread -x PublicThread` selects none.

`active` and `archived` are an independent state dimension, not children of the
`threads` group. They narrow whichever thread types survive type selection.
Consequently, `-i PublicThread -x archived` selects active public threads, but
`-i archived -x threads` selects no threads.

ID rules additionally establish hierarchical scope with nearest-target
precedence: channel/thread, parent, category, then guild. An exact included
cleanup leaf (DM, Group DM, message channel, or thread) is an explicit exception
to broad type/state and ancestor exclusions; an exact leaf exclusion still
wins. Including a guild, category, forum, media channel, or text parent does
not create such an exception for its children.

Archived cleanup always attempts to restore the original archived state. Add
`--skip-unrestorable-threads` when DMD should skip before scanning unless
restoration rights can be reasonably established.

`dmd list profiles` supports:

- `--config-path`
- `--json`
- `--quiet`
- `--verbose`
- `--redact-sensitive`
- `--redact-names` / `--no-redact-names`

## Authentication Commands

### Login

`dmd login` ensures a token is stored in the system keyring.
If a keyring token already exists, it validates that token and exits without prompting.
If an old plaintext token exists, it migrates that token to the keyring and removes the plaintext copy.
If no stored token exists, it prompts for one interactively.

Use `dmd login --replace` to prompt for a new token and replace the stored keyring token.

### Logout

`dmd logout` removes the stored token from the system keyring and deletes any old plaintext token.

### Identity Check

`dmd whoami` shows which account the active token belongs to.

## Cache Management

```bash
dmd cache clear
```

Deletes the active preserve cache file.

You can override the path with:

```bash
dmd cache clear --preserve-cache-path /path/to/cache.json
```

## Profile Management

Profile commands are useful once you want saved cleanup presets.

### Fields

`dmd profile fields` lists supported stored field names, expected value types, and whether `none` is allowed. See [Value Formats](../value-formats/) for accepted `--set` values.

### Show

`dmd profile show <name>` prints the stored profile exactly as it exists in `config.json`.

### Add

`dmd profile add <name> --set key=value ...` creates a new profile.

Examples:

- `dmd profile add nightly-dms --set include_ids=123456789012345678 --set keep_within=2w`
- `dmd profile add nightly-dms --set preserve_cache=true`

When `include_ids` or `exclude_ids` are set, profile commands validate the complete IDs through Discord before writing `config.json`.
Because that validation calls Discord, `profile add` and `profile update` also accept `--token`, `--config-path`, and retry flags.

### Update

`dmd profile update <name> --set key=value ... --unset field1 field2 ...` changes an existing profile.

Examples:

- `dmd profile update nightly-dms --set verbose=2`
- `dmd profile update nightly-dms --unset fetch_within max_messages`
- `dmd profile update nightly-dms --set max_messages=none`

### Remove

`dmd profile remove <name>` deletes a stored profile entry.

`dmd profile fields`, `show`, `add`, `update`, and `remove` support:

- `--config-path`
- `--json`
- `--quiet`
- `--verbose`
- `--redact-sensitive`

## Important Flags

### Keep Last

`--keep-last` keeps the last N messages in each channel.

Examples:

- `--keep-last 20`
- `--keep-last 0`

### Keep Last Scope

`--keep-last-scope` controls how `--keep-last` is counted.

- `all`: count all recent messages in the channel
- `mine`: count only your deletable messages

The default is `all`.

### Keep Within

`--keep-within` keeps messages and reactions newer than a given time delta, measured backwards from the time the command runs.

Examples:

- `--keep-within 2w`
- `--keep-within "1w3d"`
- `--keep-within "0"`

See [Value Formats](../value-formats/#time-deltas) for the full time delta syntax.

### Fetch Within

`--fetch-within` limits how much history is fetched from each channel, measured backwards from the time the command runs.

This improves speed, but it also means the tool only sees a slice of history unless you widen the fetch window or use preserve-cache strategically.

If a profile defines a fetch window and you want to remove it for one run, use:

- `--fetch-within none`

### Buffered Mode

`--buffer-per-channel` fetches one channel into memory before evaluation instead of walking it lazily.

Use this when you want stronger per-channel timing visibility in a manual run and do not mind the extra per-channel buffering cost.

If a profile defines a message limit and you want to remove it for one run, use:

- `--max-messages none`

### Thread Discovery

`dmd list channels` discovers all accessible announcement, public, and private
threads by default, including active and archived threads. Cleanup uses the
same thread scope.

Use these local commands to inspect the accepted filter values:

```bash
dmd list channel-types
dmd list thread-states
```

Channel-type exclusions use the canonical names shown in discovery output. Supported values are:

- `GuildText`, `GuildAnnouncement`, `GuildVoice`, and `GuildStageVoice`
- `AnnouncementThread`, `PublicThread`, and `PrivateThread`
- `DM` and `GroupDM`

Thread state is independent of channel type. Use `-x archived` to search active threads only, or `-i archived` to select archived threads.

Use `-x threads` as a broad shortcut for excluding `AnnouncementThread`,
`PublicThread`, and `PrivateThread` together. A concrete include such as
`-i PublicThread` is more specific and adds that type back. A concrete
exclusion remains authoritative at the same level.

Exact leaf IDs can override broad type/state exclusions. Existing ID hierarchy remains available for specific targets: use `--exclude <category-id>` to exclude one Discord category and its selected descendants.

Forum and media channels are containers rather than message channels. Thread discovery makes those parent IDs selectable, while cleanup operates on their discovered thread posts.

The Rich terminal tree nests each thread below its visible parent channel. If the parent is unavailable or filtered out, the thread remains at category level with parent context. JSON output keeps its existing flat channel entries and `parent_id` fields for compatibility.

Thread discovery can issue one or more paginated API requests for every
relevant thread-capable channel in each visible guild. Exclusions avoid searches
when none of the remaining thread types can exist under a parent. Archived
results use a buffered content plan and guarded activation only when that plan
contains work. `--delete-owned-threads self-only` or `all` follows the same
selected thread scope.

`dmd list channels` completes this discovery before rendering its tree. Cleanup instead advances one guild and one parent at a time: it completes a parent's paginated thread snapshot, processes that parent and its child threads, and then continues. Explicit include/exclude IDs receive exact preflight validation before deletion but reuse that preflight data and remain incremental.

### Owned Thread Deletion

`--delete-owned-threads` controls deletion of the thread channel itself. It never targets a thread whose `owner_id` differs from the authenticated user.

- `none`: never delete a thread container. This is the default.
- `self-only`: fetch the complete thread history without `--fetch-within` or `--max-messages` boundaries, then delete only when every returned message belongs to the authenticated user and the scan can be proven complete.
- `all`: permit deletion despite messages from other users. Real execution does not use authorship as an eligibility condition; dry-run still scans complete history to report impact.

Discord still requires `MANAGE_THREADS`. If Discord rejects a deletion, DMD falls back to ordinary own-message and own-reaction cleanup inside that thread.

<Aside type="caution" title="Thread deletion removes a shared container">
Successful thread deletion overrides `--keep-last`, `--keep-within`, `--keep-reactions`, fetch boundaries, and preserve-cache decisions for that thread. `all` can remove other users' messages and reactions. `self-only` excludes known foreign messages at scan time, but can still remove reactions placed by other users and cannot make the scan/delete sequence atomic. Always use a narrow thread-ID dry-run first.
</Aside>

See [Delete Owned Threads](../../guides/delete-owned-threads/) for the full safety model.

### Boolean Overrides

Boolean clean flags support explicit negative forms as well.

Examples:

- `--dry-run` / `--no-dry-run`
- `--keep-reactions` / `--no-keep-reactions`
- `--preserve-cache` / `--no-preserve-cache`
- `--buffer-per-channel` / `--no-buffer-per-channel`
- `--exclude-threads` / `--no-exclude-threads`
- `--quiet` / `--no-quiet`
- `--json` / `--no-json`
- `--redact-names` / `--no-redact-names`

This matters most when a profile enables one of these options and you want to turn it off for a single run.

### Sensitive Redaction

`--redact-sensitive` redacts sensitive values in normal CLI logs, JSON logs, and discovery output.
`-r` is the short alias.

Examples:

- `--redact-sensitive`
- `--redact-sensitive 4`
- `--redact-sensitive 0 4`
- `--redact-sensitive 4 4`
- `-r 4 --no-redact-names`

For `dmd list guilds` and `dmd list channels`, this redacts guild names, channel names, category names, DM names, and IDs.
Use `--no-redact-names` when you want names to stay visible while IDs remain redacted.

Here, "names" means Discord display names used to identify targets and accounts:

- authenticated username
- guild names
- category names
- channel names
- DM and Group DM display names

See [Value Formats](../value-formats/#redaction-values) for profile redaction values.
