---
title: Log Output
description: How to interpret v3 terminal and JSON output.
---

The CLI does not always show the same kind of output.

What you see depends mainly on:

- whether you are in `--dry-run`
- your verbosity level (`-v`, `-vv`, `-vvv`)
- whether you enabled `--buffer-per-channel`
- whether you enabled `--json`
- whether you enabled `--redact-sensitive`

## Verbosity levels

### Default

Shows the high-level run flow:

- authentication
- active deletion cutoff, when applicable
- channel processing
- per-channel summaries
- final summary

### Event Level

`-v` adds message and reaction action events:

- `Would delete message ...`
- `Would delete reaction ...`
- `Deleting message ...`
- `Deleting reaction ...`

### Detail Level

`-vv` adds detail lines:

- `Content: ...`
- `Reaction: ...`
- `Reactions: ...`

### Operational Level

`-vvv` adds operational diagnostics:

- fetch summaries
- wait timing between fetch batches
- retry/rate-limit related diagnostics
- unavailable thread-parent skips during thread discovery

## Dry-run output

In `--dry-run`, the CLI reports what it *would* do instead of deleting anything.

Each channel ends with:

- a summary line
- a timing line with:
  - `scan time`
  - `est. execute time`
  - `est. total time`

The final run summary uses the same two-line structure.

When a planned parent deletion would also remove reactions from other users,
the dry-run summary adds `foreign reactions affected` with separate normal and
Super Reaction counts. These are exact reaction instances at scan time, not
unique users. Missing, malformed, or inconsistent required reaction fields are
shown as `unknown`.

Owned-thread deletion dry-runs also show an impact line with your message count,
the other-or-unknown author count, and the foreign reaction split. The `all`
mode performs this reporting scan even though real execution does not use
authorship as an eligibility condition.

## Deletion mode output

In a real deletion run, each channel ends with:

- a summary line
- a timing line with:
  - `total time`
  - and `execute time` when buffered mode is enabled

Message and reaction summaries separate concrete delete outcomes:

- `deleted`: Discord confirmed deletion
- `absent`: Discord returned `404`; the desired state is already reached, but DMD does not claim which request or actor removed it
- `failed`: Discord confirmed that deletion could not be completed

Owned-thread summaries use the same outcomes. An absent thread is terminal, so
DMD does not attempt message cleanup inside a container Discord says no longer
exists. Transient retry exhaustion and unexpected responses abort the run
instead of being counted as ordinary failed actions.

## Buffered mode

With `--buffer-per-channel`, the CLI fetches one channel up front before evaluating it.

This is most useful when you want stronger per-channel timing visibility in a manual run.

## Meaning of the timing fields

- `scan time`: how long it took to fetch and evaluate the channel or run
- `est. execute time`: rough estimate for how long delete/reaction actions would take
- `est. total time`: `scan time + est. execute time`
- `execute time`: measured action time during a real buffered deletion run
- `total time`: actual wall-clock time for that executed channel

The estimates are rough. They are based on the configured pacing intervals between planned delete/reaction actions, not exact future API latency or Discord bucket waits.

## JSON mode

Use `--json` when you want machine-readable output instead of terminal-oriented log lines.

In `--json` mode:

- terminal-oriented formatting is disabled
- each log entry is emitted as a JSON object
- entries still include timestamp, level, logger, and message
- cleanup action, decision, and summary entries additionally include a stable
  `event` name and scalar `data` fields

Structured cleanup fields describe modes, artifact/action kinds, and counts.
They do not contain Discord IDs, names, message content, emoji names, local
paths, or credentials. The human-readable `message` field still follows the
normal sensitive-value redaction settings.

## Sensitive value redaction

Use `--redact-sensitive` when you want to share logs without exposing private names, IDs, or local paths.
Use `--no-redact-names` when you want human-readable names to stay visible while IDs and other sensitive values remain redacted.

When redaction is enabled:

- IDs use the configured prefix/suffix window
- human-readable sensitive values are fully hidden as `***` by default
- human-readable names remain visible when `--no-redact-names` is set

Names means Discord display names used to identify targets and accounts: authenticated username, guild names, category names, channel names, and DM / Group DM display names.
It does not include message content, local paths, tokens, or emoji names.

This applies to normal CLI logs, JSON logs, and discovery output from `dmd list guilds` or `dmd list channels`.
